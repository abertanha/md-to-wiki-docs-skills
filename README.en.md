# md-to-wiki — Multi-Format Spec Publisher

Converts markdown files generated by _spec-driven_ sessions (tlc-spec-driven, ai-harness-engineer, etc.) into documentation in your chosen format.

## Supported Formats

| Format | Description |
|--------|-------------|
| **Pure HTML (MkDocs Material)** | Static site with search, dark/light mode, GitHub Docs look |
| **Swagger / OpenAPI** | Parses API contracts from markdowns, generates OpenAPI 3.0 spec, serves with Swagger UI or ReDoc |
| **GitHub Tab Wiki** | Publishes markdowns directly to the repository's GitHub Wiki tab |
| **DokuWiki** | Converts to DokuWiki syntax, outputs a ready-to-import directory |
| **PDF** | Compiles all markdowns into a single PDF document |

## Workflow

1. **Onboarding** — The skill interviews you to understand audience, goals, scope, which files to include, and whether to link related GitHub issues/PRs
2. **Issues/PRs fetch** — Scans markdowns for `#123` references, asks if you have more, then fetches via `gh` CLI (fallback `curl`). Caches locally. Only the main repo is fetched; cross-repo refs are mentioned.
3. **Format recommendation** — Suggests the best format based on your answers
4. **Execution** — Generates the documentation in the chosen format, appending a **References appendix** with fetched issues/PRs at the end
5. **Deploy** — Offers deployment options (GitHub Pages, local server, etc.)

## Hard Rules

- **No ASCII diagrams.** Every flow, architecture, sequence, or state must use Mermaid (` ```mermaid ... ``` `)
- ASCII diagrams found in source markdowns are automatically converted to Mermaid
- **Issues are cached forever** — re-fetch only if the user explicitly asks

## OS Detection

The OS is detected **once** during the version check step (step 1a). All subsequent script calls reuse the same variables:

| OS / Environment | `SCRIPT_EXT` | `SCRIPT_RUNNER` | Behavior |
|------------------|-------------|-----------------|----------|
| Linux / macOS | `.sh` | (empty) | Runs `.sh` scripts directly |
| Git Bash (MINGW/MSYS/CYGWIN) | `.sh` | (empty) | Runs `.sh` scripts natively (bash available) |
| Pure PowerShell | `.ps1` | `powershell -File` | Runs `.ps1` scripts via PowerShell |

Script calls use the pattern:
```bash
$SCRIPT_RUNNER "$SKILL_DIR/scripts/<name>$SCRIPT_EXT" <positional_args>
```

All scripts accept **identical positional arguments** across `.sh` and `.ps1` — no flags needed, no OS-specific syntax.

## Audience Values

The `generate-index.sh` and `generate-index.ps1` scripts accept the following audience values (singular and plural both work):

| Value | Sections included |
|-------|------------------|
| `developer`, `developers`, `devs` | Quick Start, Feature table, Architecture, Getting Started, Development |
| `stakeholder`, `stakeholders` | Overview with Roadmap, Feature table, Architecture, Getting Started |
| `general` (default) | Overview table, Feature table, Architecture, Getting Started |

## Installation

### Option 1 — Clone and symlink

```bash
git clone <repo-url>
ln -s "$(pwd)/md-to-wiki" ~/.config/opencode/skills/md-to-wiki
```

### Option 2 — Direct copy

```bash
git clone <repo-url>
cp -r md-to-wiki ~/.config/opencode/skills/md-to-wiki
```

### Option 3 — Load via URL (opencode only)

Add to your `opencode.json`:
```json
{
  "skills": {
    "urls": ["https://raw.githubusercontent.com/<user>/<repo>/main/skills/md-to-wiki/SKILL.md"]
  }
}
```

### Target locations

| Scope | Path |
|-------|------|
| Global (all sessions) | `~/.config/opencode/skills/md-to-wiki/` |
| Per project | `.opencode/skills/md-to-wiki/` |
| Cursor skills | `.cursor/skills/md-to-wiki/` or `.cursor/skills-cursor/md-to-wiki/` |

## Per-Format Dependencies

- **Pure HTML:** `pip install mkdocs mkdocs-material`
- **Swagger:** Node.js + npx (or just a browser)
- **GitHub Wiki:** git
- **DokuWiki:** `sudo apt install pandoc` (Linux) or `winget install pandoc` (Windows)
- **PDF:** `pip install weasyprint` + `sudo apt install pandoc`

## Windows / PowerShell Compatibility

All scripts ship with both `.sh` (Linux/macOS) and `.ps1` (Windows PowerShell) versions. The skill auto-detects the OS and runs the correct script. On Windows:

- **Git Bash** (MINGW/MSYS/CYGWIN) runs `.sh` scripts directly — same as Linux
- **Pure PowerShell** runs `.ps1` scripts — WSL is **not required**
- The `gh` CLI works on Windows too (install via `winget install GitHub.cli`)
- Pandoc on Windows: `winget install pandoc`
- mkdocs works via `pip install mkdocs mkdocs-material` on Windows same as Linux

## Usage Examples

During a session, say:

- "Build wiki from specs"
- "Publish specs as HTML site"
- "Generate GitHub Wiki docs"
- "Export to DokuWiki"
- "Generate PDF from specs"
- "Diagram this architecture" → triggers the Mermaid rule

## Multi-Directory Input

You can specify custom directories containing markdown files in addition to (or instead of) `.specs/`. The skill will scan all specified directories and merge the results.
